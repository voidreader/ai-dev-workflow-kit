# Unity 아키텍처 어드바이저

게임플레이 스크립트를 대량 생성하기 전,
또는 더 깔끔한 아키텍처가 필요할 때 먼저 사용하세요.

## 워크플로우 → 출력 매핑

각 단계의 결과가 출력 항목에 직접 연결됩니다.

| 단계 | 하는 일 | 출력 항목 |
|---|---|---|
| 1 | 고통 지점과 변경 빈도 파악 | **현재 상태 진단** |
| 2 | 핵심 루프와 최소 시스템 정의 | **핵심 루프 요약** |
| 3 | 레이어 분리 및 모듈 경계 확정 | **권장 모듈 (3~7개)** |
| 4 | 데이터 소유권 결정 | **데이터 소유권** |
| 5 | 통신 방식 결정 | **통신 규칙** |
| 6 | 지금 할 것 / 지금 건너뛸 것 명시 | **Do now / Skip now** |

---

## 단계별 판단 기준

### 1. 현재 상태 진단
아래 신호를 확인한다.

- 같은 코드를 3곳 이상 복붙하고 있다 → 추상화 시점
- 기획이 주 1회 이상 바뀐다 → 구조화 보류
- 전체 흐름을 머릿속에 못 담게 됐다 → 모듈 분리 시점
- 특정 클래스만 계속 건드린다 → 책임 분리 시점

### 2. 핵심 루프 요약
런타임에 반드시 있어야 하는 시스템만 정의한다.
"있으면 좋은" 시스템은 이 단계에서 제외한다.

### 3. 권장 모듈
레이어 분리 기준:
- 씬 / 부트스트랩: 컴포지션과 초기화만 담당
- 게임플레이 / 도메인: 순수 C# 클래스 우선
- 데이터 / 설정: ScriptableObject (런타임 상태 금지)
- 뷰 / 프레젠테이션: MonoBehaviour는 여기서만 두껍게 허용

### 4. 데이터 소유권
| 데이터 종류 | 위치 |
|---|---|
| 제작자가 직접 편집하는 설정값, 공유 상수 | ScriptableObject |
| 게임 실행 중 변하는 상태 | 순수 C# 클래스 |
| 씬에만 존재하는 고유 참조 | MonoBehaviour 필드 |

### 5. 통신 규칙
| 상황 | 방식 |
|---|---|
| 같은 모듈 내부 | 직접 참조 |
| 모듈 간, 호출 방향이 명확할 때 | 인터페이스 |
| 모듈 간, 호출 방향이 불명확할 때 | 이벤트 |
| 실행 순서 보장이 필요할 때 | 커맨드 |

### 6. Do now / Skip now
- **지금 할 것**: 현재 고통 지점에서 신호가 보이는 것만
- **지금 건너뛸 것**: 신호가 없으면 추상화 금지. 이유를 명시할 것

## 판단 원칙

- `MonoBehaviour`는 컴포지션 연결 다리로만 얇게 유지
- 재사용 가능한 게임플레이 규칙은 순수 C# 클래스 우선
- `ScriptableObject`는 제작자가 편집하는 설정값에만 사용, 실행 중 상태를 담는 용도 금지
- 의존성은 명시적으로. 전역 상태는 규모가 명확히 정당화할 때만 허용
- 교과서적 SOLID 표현보다 지금 고통을 줄이는 구조 우선
- 인상적인 아키텍처보다 반복 가능한 작은 구조 우선
- UnityCatClicker에서는 Manager 접근을 반드시 `Managers.XX` 통해서만 한다. Manager 직접 인스턴스화나 MonoBehaviour 상속(GameManager 예외)을 하지 않는다.
